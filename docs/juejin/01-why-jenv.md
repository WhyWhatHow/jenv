# 曾经淋过雨,现在想为你撑伞:jenv 诞生记

## 那个被 JDK 折磨的夜晚

大一的时候,有一个周日的深夜,我永远忘不了。

那天晚上,我需要完成 Java 课程的第一个作业——一个简单的 Hello World 程序。但是,在写代码之前,我必须先装上 JDK。

我按照网上的教程,下载了 JDK 安装包,双击安装,然后就是噩梦的开始:配置环境变量。

看着那一串串陌生的路径 `C:\Program Files\Java\jdk1.8.0_291\bin`,我小心翼翼地复制到 `JAVA_HOME` 和 `PATH` 里。但是,**我根本不知道自己配对了没有**。

打开 CMD 窗口,输入 `java -version`,回车——没反应。

我的心一紧:是不是哪里配错了?会不会把系统搞坏了?

于是我又回去检查环境变量,改了又改,重启电脑,再试,还是不行。就这样,**我在恐惧、无助和崩溃中度过了整整 2 个小时**,经历了无数次 reboot。

那一夜,我对着黑色的 CMD 窗口,真的快哭了。

## 新手的三重恐惧

后来我发现,我不是一个人。几乎每个 Java 新手都经历过类似的恐惧:

### 恐惧 1:看不懂的 JAVA_HOME 路径

`C:\Program Files\Java\jdk1.8.0_291` 这是什么?为什么有这么多斜杠?为什么有的教程说要加 `\bin`,有的又说不加?

对于新手来说,这些路径就像天书,每一步都在猜,每一步都怕错。

### 恐惧 2:不知道改对了没有

改完环境变量,你不会立刻得到反馈。你必须重启终端,甚至重启电脑,然后试一下 `java -version`。

如果不行,你又要回去改,再重启,再试。这种**缺乏即时反馈的折磨**,让人怀疑人生。

### 恐惧 3:害怕一步错,全盘崩

"如果我把系统环境变量改错了,电脑会不会坏掉?"

"如果我装了多个 JDK 版本,会不会冲突?"

"如果我不小心删了什么,怎么办?"

这种**对未知的恐惧**,让新手每一步都走得小心翼翼,战战兢兢。

### 对比:Linux/macOS 的幸福

更让人难过的是,我后来发现 Linux 和 macOS 用户有 `sdkman` 这样的神器,可以一键安装、一键切换 Java 版本。

而 Windows 用户呢?我们只能手动改环境变量,忍受这种原始的折磨。

**Windows 用户,难道就该是二等公民吗?**

## 转折:发现 jenv-for-windows 的希望与失望

大二的时候,我无意中发现了 [jenv-for-windows](https://github.com/FelixSelter/JEnv-for-Windows) 这个项目。

当时我眼前一亮:**终于有人做 Windows 的 Java 版本管理工具了!**

我兴冲冲地下载下来,按照教程配置好,然后在终端输入 `jenv use java8`,切换版本。

然后,我等了 1 秒、2 秒、3 秒......

`java --version` 居然需要 **1-2 秒**才能出结果!

我以为是我的电脑太慢,但换了几台机器试,都是这样。后来我才知道,这是因为 jenv-for-windows 使用了批处理脚本包装 Java 命令,每次调用都要经过一层转发,所以速度很慢。

**每次运行 Java 命令都要等 1-2 秒,这还怎么写代码?**

我陷入了沉思:为什么 Linux/macOS 的工具都这么流畅,而 Windows 用户就要忍受这种卡顿?

**难道,就没有一个真正为 Windows 优化的 Java 版本管理工具吗?**

## 决心:我要给 Windows 用户一个答案

那天晚上,我想起了大一那个被 JDK 折磨的夜晚。

我想起了那种恐惧、无助、崩溃的感觉。

我想起了无数次重启电脑,无数次对着 CMD 窗口发呆。

**曾经淋过雨,现在我想为 Java rookie 撑一把伞。**

我决定:自己做一个 Java 版本管理工具,专门为 Windows 用户优化,让新手可以 **happy coding**,把时间花在有意义的地方,而不是浪费在配置环境上。

### 我的目标很简单:

1. **速度要快**:扫描、切换都要在 300ms 内完成,不能让用户等待
2. **操作要简单**:几个命令就能搞定,不需要手动改任何配置文件
3. **新手友好**:自动检测、智能提示、清晰的错误信息
4. **Windows 优化**:不是 Linux 工具的移植,而是从第一天就为 Windows 打造

**Windows 用户不应该是二等公民,我们也应该有一流的开发体验。**

## 行动:从零到 v0.6.7

但是,我遇到了一个问题:**我不会 Go 语言。**

如果是传统的开发方式,我需要先花几个月学 Go,然后才能开始写代码。

但是,我生活在 2024 年,一个 **AI 协作开发的时代**。

我决定边学边做,使用 Cursor、Claude、Gemini 等 AI 工具,和它们一起开发 jenv。

### 开发历程:

- **第 1-2 周**:用 Cursor 快速搭建基础功能(add/list/use/remove)
- **第 3 周**:和 OpenAI GPT-4 讨论架构优化,重构代码
- **第 4 周**:用 Trae 美化 CLI 界面,添加主题系统
- **第 5-6 周**:用 Gemini 优化性能,实现 Dispatcher-Worker 并发模型

经过 6 周的努力,jenv v0.6.7 诞生了。

### 成果:

- **扫描速度**: 3s → **300ms** (90% 性能提升)
- **切换速度**: 立即生效,无需重启
- **用户体验**: 彩色输出、主题切换、智能提示
- **跨平台支持**: Windows(完整)、Linux(完整)、macOS(进行中)

![jenv 演示](../../assets/jenv.gif)

**从不会 Go,到做出一个 10 倍更快的工具,我只用了 6 周。**

这就是 AI 时代的力量。

## 结尾:为你撑伞

技术应该让生活更简单,而不是更复杂。

Java 环境配置,本来就不应该是一个让新手恐惧的障碍。

我创建 jenv,就是希望:

- **Java 新手**可以把时间花在学习 Java 本身,而不是折腾环境
- **Windows 用户**可以享受和 Linux/macOS 一样流畅的开发体验
- **每一个开发者**可以专注于创造,而不是被工具困扰

曾经,大一的我在深夜被 JDK 折磨得快哭了。

现在,我想为每一个 Java rookie 撑一把伞。

**如果 jenv 能帮你节省 1 小时的配置时间,让你少经历一些恐惧和崩溃,那我就觉得这一切都值得了。**
